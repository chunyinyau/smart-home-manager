Project Context: Wattch1. Project OverviewWattch is a Smart Home Energy Management system designed to help residents in Singapore and small offices keep their monthly electricity bills under budget.The system achieves this through four primary mechanisms:Logging meter readings.Forecasting budget risk.Automated control over non-essential appliances.Interactive Telegram updates to the user.2. Core Business ScenariosScenario 1: Meter Reading & Immediate ResponseA resident takes a photo of their meter and sends it to the AI Agent via Telegram.The system utilizes an OCR API to extract the meter reading value.The budget plan is updated.If the risk level is deemed critical, the system automatically turns off non-essential appliances.A Telegram notification and warning are sent to the user.Scenario 2: High-Drain Appliance & User DecisionThe system detects anomalous high energy usage from a specific appliance.A re-forecast is triggered, elevating the budget risk level to HIGH.The system sends an interactive Telegram alert (e.g., "Water Heater on past 4 hours. Reply /keep or /off.") with a defined Time-To-Live (TTL).Path A: If the user replies /off before the TTL expires, the appliance is shut down.Path B: If the TTL expires with no user reply, the system automatically cuts the appliance off.Scenario 3: Month-End Staged Budget RescueA scheduled day 26 cron trigger evaluates the month-end forecast.Monthly meter readings are aggregated to calculate any projected overrun.The system executes a staged rescue plan, re-forecasting after each step:Stage A: Cut high-usage non-essential appliances.Stage B: If still over budget, cut medium-usage non-essentials.Stage C: If still over budget, lock essential appliances and escalate the issue to the user.3. Service-Oriented Architecture (SOA)The system is built on a distributed Service-Oriented Architecture where each major capability is a separate service with a clear contract. Services interact through defined interfaces rather than direct shared function calls.Interaction FlowUser -> Telegram UI -> Bot Gateway / PicoClaw OrchestratorThe Bot Gateway routes requests to the following independent microservices:ApplianceServiceBudgetServiceForecastServiceAutomationServiceProfileServiceHistoryServiceRateServiceMicroservice Responsibilities & Database MappingApplianceService: Owns appliance state, priority, on/off actions, and usage data. Connects to Appliance DB.RateService: Owns tariff lookups (via SP energy or mock data). Connects to Rate DB.BudgetService: Owns the monthly budget cap, currently spent amounts, and risk messaging. Connects to Budget DB.ForecastService: Owns the budget forecasting logic, utilizing history and profile data.AutomationService: Owns prioritization and shutdown decisions. Orchestrates calls to ApplianceService, BudgetService, and ForecastService.HistoryService: Owns previous monthly bill records. Connects to History DB.ProfileService: Owns user flat/apartment settings and baseline profile data. Connects to Profile DB.DashboardService: Acts as a composition layer for the Web Dashboard UI by aggregating data from the underlying atomic services.4. AI Integration PointsTo maintain a strict boundary for AI execution, only a small number of specific services are permitted to call the PicoClaw AI API:ForecastService -> PicoClaw AIAutomationService -> PicoClaw AI
